# Quick Task 260331-i16: Fix 502 NATS No-Responders Error When Sending Messages

## One-liner

Structured provider error classification in the Rust gateway backend, frontend detection utilities, and a dismissible warning card with actionable guidance on the OpenClaw webapp page.

## What Was Done

### 1. Rust Backend: Provider Error Classification (`gateway_ws.rs`)

Added `classify_gateway_response()` helper function that inspects gateway JSON responses for:
- "no responders" or "nats" text in the response body
- HTTP 502/503 status codes in error objects
- Nested `/error/message` with provider/NATS keywords
- `server_error` type with provider/upstream/backend context

Modified `gateway_ws_call` to check the response via this classifier before returning `Ok`. When a provider error is detected, returns `Err("PROVIDER_ERROR: {message}")` instead of `Ok(raw_json)`, enabling frontend pattern matching.

### 2. Frontend Utility: Provider Error Detection (`provider-errors.ts`)

Created `src/lib/provider-errors.ts` with three exports:
- `isProviderError(error)` — detects the `PROVIDER_ERROR:` prefix and raw pattern fallbacks (no responders, nats, 502, 503)
- `getProviderGuidance()` — returns actionable steps: check API keys, verify model provider, ensure service reachable, restart gateway
- `extractProviderErrorMessage(error)` — strips the prefix and returns the human-readable message

### 3. Hook Integration (`use-gateway.ts`)

- Enhanced `useGatewayCall` with `wrapProviderError()` that throws a structured `ProviderError` with `isProviderError: true`, `providerGuidance`, and `providerMessage` fields when a provider error is detected
- Added `useGatewayMutation<TData, TVariables>` hook for mutation-based gateway calls (chat completions, config patches) with the same provider error detection
- Exported `ProviderError` interface for consumers to type-check provider errors

### 4. Warning Card UI (`openclaw-webapp.tsx`)

- Added dismissible `Alert` (warning variant) with `AlertTriangle` icon that shows when a provider error is detected
- Displays the error message and all guidance steps in a list
- Added "Check Provider Status" button that makes a lightweight `config.get` call to test provider reachability
- Non-blocking: "Open Control UI" button remains functional even with the warning displayed
