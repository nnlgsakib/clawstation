## Objective

Fix the 502 NATS "no responders" error that surfaces as a raw, unhelpful error when users send messages in the OpenClaw Control UI. Add provider error classification in the Rust backend and user-friendly error display with actionable guidance in the frontend.

**Purpose:** Users hitting provider errors get zero guidance. They see raw JSON error blobs with no indication of what's wrong or how to fix it.
**Output:** Classified provider errors + warning UI on the webapp page.

    - "no responders" text

    If detected, return `Err(...)` with a classified prefix: `"PROVIDER_ERROR: {details}"` instead of `Ok(raw_json)`. This lets the frontend pattern-match on the prefix.

    Also add a `classify_gateway_response` helper function that inspects the JSON value and returns `Option<String>` with the error message if it's a provider error. Keep it simple — string pattern matching on serialized JSON.

    Example detection logic:
    ```rust
    fn classify_gateway_response(response: &serde_json::Value) -> Option<String> {
        let text = response.to_string().to_lowercase();
        if text.contains("no responders") || text.contains("nats") {
            return Some("Inference provider is not responding. The configured model provider may be down or misconfigured.".to_string());
        }
        if let Some(status) = response.get("error").and_then(|e| e.get("http_status")).and_then(|s| s.as_u64()) {
            if status == 502 || status == 503 {
                return Some(format!("Provider returned HTTP {status}. Model provider may be unavailable."));
            }
        }
        // Check nested error patterns
        if let Some(err_msg) = response.pointer("/error/message").and_then(|v| v.as_str()) {
            if err_msg.contains("no responders") || err_msg.contains("NATS") || err_msg.contains("provider") {
                return Some(format!("Provider error: {err_msg}"));
            }
        }
        None
    }
    ```
